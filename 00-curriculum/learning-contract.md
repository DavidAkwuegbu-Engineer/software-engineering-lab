# Learning Contract

## Mission

Become a software engineer capable of building secure, useful, polished systems and innovating in the software world.

## How This Lab Should Train You

This is not a passive notes folder. It is a working lab for building skill through repetition.

Each lesson should include:

- A clear engineering outcome.
- A small implementation task.
- Tests or verification commands.
- Debugging practice when something fails.
- Security considerations.
- Evidence that can later support a portfolio.
- Git commits that show steady, understandable progress.

## Muscle Memory Loop

Use this loop every session:

1. Read the task and identify the expected behavior.
2. Read the existing code before editing.
3. Make the smallest useful change.
4. Run the test or check.
5. Fix the first failure.
6. Record what changed and what was verified.
7. Commit the completed unit of work with a clear message.

## Standards

- Prioritize secure defaults.
- Validate untrusted input.
- Avoid unnecessary dependencies.
- Prefer readable code over clever code.
- Leave every lesson cleaner and easier to continue than it was before.
- Do not move on from a lesson until the verification step is honest.
- Never commit secrets, tokens, `.env` files, or generated dependency folders.
